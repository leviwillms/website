## Findings

Over the last couple weeks I have greatly enhanced the Svelte frontend for the thesis. Some of the things I've done include:

1. Removing the frontend console for querying alignment.
2. Adding auto-scroll to show source images on the left and resultant markdown on the right.
3. Rendering all images together instead of individual image selection.
4. Showing bbox alignment on hover from either side (left or right) of the display.

However, following these changes I've noticed some glaring issues with our alignment algorithm. And, without having some kind of evaluation metric, there is no way for us to judge whether the algorithm is _correctly_ doing alignment between source and result.

Given our example paper, _Deep Residual Learning for Image Recognition_, we can draw the following conclusions from the alignment [attached here](./alignment.json):

1. **word_611 crosses the paragraph boundary**
   Its char_span [2553, 2612] literally spans from "y deep models.\n\n" (end of "greatly benefited" paragraph) into "Driven by the significance of depth, a ques" (start of next paragraph). This single word node's bounding box is physically on the image somewhere, but its alignment covers text from TWO separate markdown paragraphs. When you hover either paragraph, this word gets included.

   This is just one example of words crossing paragraph boundaries.

2. **Mega-words: word_682 (217 chars) and word_683 (177 chars)**
   These single "word" entries cover entire sentences. word_682 covers "g this question was the notorious problem of vanishing/exploding gradients [1, 9]...initialization [23, 9, 37," — that's 217 characters of markdown mapped to ONE layout word's bounding box. On the image, this word's bounding box is a small rectangle around one actual word, but its alignment says it "owns" 217 characters of text. This means its bounding box lights up whenever you hover ANY paragraph that overlaps those 217 chars.

3. **Only 12 word entries for a 531-character paragraph**
   The "Driven by" paragraph has ~80 words but only 12 alignment entries with non-zero char_spans. And of those 12, two are mega-words covering most of the paragraph. So you get very few, very large bounding boxes instead of precise per-word boxes.

Verdict: This is a data quality issue in the alignment algorithm, not a frontend bug. The alignment in ppx-dev is producing word-level entries where single OCR words get mapped to enormous char_spans (hundreds of characters). The frontend code is working correctly — it finds all word nodes overlapping the hovered range, but the alignment data is too coarse/imprecise to produce good results.

The fixes would be in the ppx-dev alignment pipeline — either improving the alignment algorithm's granularity or post-processing to split mega-word entries.

## Evaluation Plan

As stated in the findings, we have no way to deterministically evaluate the alignment. With that core issue in mind I now must return to the benchmarking idea I had on `March 11th, 2026`.

This idea borrows scraping techniques from the PubLayNet dataset to reference the _PMCOA (PubMed Central Open Access)_ dataset of medical papers. Since these are born-digital documents they have the XML format readily available. Using PDFMiner, we can source the textbox's bonding box locations and use different heuristics to derive lines and regions.

### 4-Level Hierarchy: Sources and Derivation

Different than my ideas of March 11th, a relative mapping can be derived as follows:
| Level | Source | How |
|---|---|---|
| `region` | PubLayNet COCO JSON | Pre-annotated — bbox + label already done |
| `block` | PDFMiner textbox splits at XML node boundaries | Re-run PDFMiner; split where XML nodes change mid-textbox |
| `line` | PDFMiner textlines | Direct output of PDFMiner within each textbox |
| `word` | Tokenize each textline's XML text | `word_tokenize(xml_text_for_line)` |
| `order` | NLM XML document sequence | Propagated via text matching; correct for multi-column layouts |

### Tools

- **[pdfminer.six](https://pdfminersix.readthedocs.io/)** — parse PDFs, extract `LTTextBox` and `LTTextLine` with bboxes
- **[pdfplumber](https://github.com/jsvine/pdfplumber)** — higher-level wrapper around pdfminer, easier word-level extraction
- **[PyMuPDF (fitz)](https://pymupdf.readthedocs.io/)** — alternative PDF parser, faster, good for word-level bbox extraction

---

### Ground Truth Annotation Format

I am proposing one JSON file per document page:

```json
{
  "doc_id": "PMC1234567_page1",
  "image": "samples/PMC1234567_page1.png",
  "tokens": ["Attention", "Is", "All", "You", "Need", "The", "dominant", "..."],
  "ground_truth": {
    "region_0": { "span": [0, 5], "label": "title" },
    "region_1": { "span": [5, 310], "label": "text" },
    "block_0": { "span": [5, 61], "order": 0 },
    "block_1": { "span": [61, 120], "order": 1 },
    "line_0": { "span": [5, 18] },
    "line_1": { "span": [18, 31] }
  }
}
```

Key conventions:

- **Half-open `[start, end)` intervals** — consistent with Python slicing and algo output
- **`tokens` stored explicitly** — ensures reproducibility if `word_tokenize` behaviour ever changes
- **Node IDs match `build_layout_tree` output** — so `TreeAlignment.align` can be directly compared against `ground_truth`

## Plan in practice

### Results

#### Overall (all levels combined)

| Metric         | Greedy   | **DP**       | Hybrid   |
| -------------- | -------- | ------------ | -------- |
| Token F1       | 0.869    | **0.887**    | 0.869    |
| Exact match    | 47.6%    | **54.5%**    | 47.6%    |
| Boundary error | 32.5 tok | **30.3 tok** | 32.5 tok |

**DP wins across the board. Hybrid matches Greedy exactly** — this means the token sequences are long enough (>100 tokens) on every node, so `hybrid_align_nodes` always falls back to Greedy. The hybrid threshold (100 tokens) needs tuning.

#### By level

| Level  | n   | Algo   | F1        | Exact   | Boundary err |
| ------ | --- | ------ | --------- | ------- | ------------ |
| region | 5   | all    | 0.619     | 0%      | 359 tok      |
| block  | 90  | Greedy | 0.832     | 23%     | 11.4         |
| block  | 90  | **DP** | **0.840** | **32%** | **10.9**     |
| line   | 428 | Greedy | 0.880     | 53%     | 33.2         |
| line   | 428 | **DP** | **0.900** | **60%** | **30.5**     |

**Region matching is broken** (F1=0.619, 0% exact, boundary error 359 tokens) — this is the known containment-based matching issue where gold regions are small per-section bboxes vs PaddleX's single large body region.

#### Key takeaways

1. **DP consistently outperforms Greedy** at both block and line levels (~1-8pp improvement in exact match)
2. **Hybrid = Greedy in practice** — the 100-token threshold is too low; every node exceeds it
3. **Line level is the most meaningful signal** (428 pairs): DP F1=0.900, exact=60%
4. **Region evaluation needs a different matching strategy** (containment-based rather than IoU)
5. Pages 8/9 have very few matched pairs due to large tables dominating those pages

### What went exactly as planned

- **Matching algorithm** — The PDFMiner → NLM XML fuzzy matching with adaptive Levenshtein tolerance (d_max) worked well. Sequential scanning with full lookahead achieved ~95%+ match rates on body text pages.

- **Evaluation metrics** — Token F1, exact match, and boundary error were implemented exactly as specified.

- **Annotation format** — The half-open `[start, end)` span convention and the gold_alignment.json structure closely match the plan.

Tooling — pdfminer.six for PDF parsing, fuzzysearch for fuzzy matching — both used as planned.

### What the plan underestimated

1. The circular ground truth problem
   The plan assumed PaddleX output could serve as the reference. It couldn't — you caught this. The benchmark was testing PaddleX alignment against itself. The fix (PDFMiner + NLM XML independently) was correct but added significant complexity.

2. Non-text content contamination
   The plan didn't account for sidebars, tables, figures, and headers polluting the markdown. These elements:
   - Aren't in TEXT_TAGS (no NLM XML nodes to match against)
   - Confuse the fuzzy search cursor when they appear in the markdown between body text paragraphs
   - Required an entire masking pipeline (mask-regions) that wasn't in the original plan at all 3. PaddleX's masking-resistant detection
   - After masking tables/figures to clean up the markdown, PaddleX re-detected them from border lines and whitespace patterns. Required tracking masked_bboxes and suppressing re-detected blocks — none of this was anticipated.

3. Coordinate system mismatch
   PDFMiner uses bottom-up points; PaddleX uses top-down pixels. The scale was also wrong initially (img_width / page_height instead of img_height / page_height). Led to zero spatial matches until caught.

4. Region-level matching is still broken
   The plan describes regions as pre-annotated PubLayNet COCO bboxes. In practice we built gold regions from NLM XML sections (one per <sec> element), which are small. PaddleX produces one large body region per page. IoU is always near zero — the plan's IoU-based matching doesn't apply at region level without a containment-based alternative.

5. Hybrid = Greedy in practice
   The plan implicitly assumed hybrid would provide a meaningful middle ground. In reality every node exceeds the 100-token threshold so hybrid always routes to greedy. The threshold needs calibration against actual token sequence lengths.

### What the plan didn't cover at all

- The masked_regions.json persistence problem: after one masking run overwrites blocks.parquet, subsequent runs can't re-derive what was masked
- Line orphaning after block removal, and the targeted filtering via masked_regions.json
- Figure/table block labels differing from what was expected (image, chart vs figure)
- The gold standard needing to be rebuilt every time the masked markdown changes

### What Diverged From the Plan?

**Region level** — Gold regions were derived from NLM XML `<sec>` elements matched to PDFMiner bboxes, not PubLayNet COCO annotations. This created a structural mismatch: gold regions are small per-section bboxes while PaddleX produces one large body region per page. IoU-based matching fails at this level and region evaluation remains an open problem.

**Annotation format** — The implementation diverged from the single-file JSON design. Gold standard data is stored as two files per page: `gold_tree.parquet` (node bboxes and hierarchy) and `gold_alignment.json` (token spans). The explicit `tokens` array was also dropped — token spans are computed at runtime against the masked markdown rather than stored.

- `gold_tree.parquet` — node bboxes and hierarchy, one row per node:
  - node_id | level_index | level_name | parent_id | x0 | y0 | x1 | y1 | label | content

- `gold_alignment.json` — token spans per node:

```json
{
  "align": {
    "gold_region_0": { "token_span": [0, 526], "char_span": [0, 2850] },
    "gold_block_0": { "token_span": [83, 86], "char_span": [412, 432] },
    "gold_line_0": { "token_span": [83, 86], "char_span": [412, 432] }
  }
}
```

### Known Limitations

I've intentionally skipped word-level evaluation. In the `evaluate.py` script, the `body_subtree()` explicitly filters to only `{"region", "block", "line"}`. This was done because:

1. **DP timeout** — the full layout tree including word nodes has 1000+ nodes, which caused dp_align_nodes (O(nL²)) to time out at 120s
2. **Data quality** — the "mega-word" problem I documented (single OCR words mapped to hundreds of characters) makes word-level gold standard unreliable.

Given this gap, the following would be required to close the gap:

- A faster DP implementation
- Or evaluating words independently per-block (smaller subproblems) rather than the full tree.
- Using NLM XML as the authoritative text for each matched line and deriving the word tokens from each line.
