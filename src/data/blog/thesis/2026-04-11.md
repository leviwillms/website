# A Realization

Published: true

## A Realization

The first step to benchmarking the correctness of the algorithm is to understand its correctness. That was the goal of April 8th's work session - to create a ground truth dataset such that we can evaluate how correct our alignment algorithm is.

After creating a pipeline using PDFMiner, XML, and PubLayNet's modified Levenshtein distance algorithm there came a realization that I was just recreating the Visual Tokens from PaddleOCR output with a different pipeline method. Essentially, I was repeating the work of benchmarking PaddleOCR while aligning the _correct_ Visual Tokens to the OCR'd Markdown. The alignment algorithm produces some type of the following:

```
Region --> MD Section
Block --> MD Section (token span)
Line --> MD Token span
Word --> MD Token span
```

At every level we are mapping the layout tree to markdown. So if we use a different method to create a ground-truth layout tree, we are _still_ mapping it to markdown. Now, after careful consideration I've come to the complete realization that _we cannot evaluate the alignment algorithm for correctness without the manual creation of a ground truth dataset._ Which, by my research, no existing dataset satisfies this requirement, and the creation of one constitutes a research effort in its own right.

With this discovery in mind I must pivot again with the framing of my thesis.

## The limitations + failure mode framing

I must change the framing to acknowledge the limitations of our work and propose a failure mode. This is the stronger near-term contribution, and it's defensible:

The genuinely novel claim isn't "our alignment is accurate" — it's "we have identified that alignment correctness in visual document OCR pipelines is fundamentally unverifiable with existing methods, and here is why." That epistemological contribution is real and useful to the community.

The paper structure that follows naturally:

- Define the visual token alignment problem precisely — what it is, why it matters for downstream tasks
- Show why existing ground truth methods (human bbox annotation, PDFMiner+XML) don't extend to alignment verification
- Show why high-accuracy OCR proxies aren't honest ground truth substitutes
- Failure mode taxonomy — where does alignment break, what are the structural causes (multi-column, tables, figures, hyphenation, OCR noise)
- Open problems — what would a real benchmark require, what the 1000-paper annotation effort would actually need to specify

That's a paper that advances the field's understanding of its own measurement limitations, which is a meaningful contribution — especially if, as you say, this specific framing of the problem is new.

## What can't any other method resolve this problem?

Methodologies that were discussed in the course of correctness evaluation were

- Building on PubLayNet's PDFMiner+XML pipeline
- Human bbox annotation
- High-accuracy OCR proxy (MistralOCR, Gemini, etc.)

None which had the capability to create a validated ground-truth for evaluation. PDFMiner+XML has a limitation of Markdown. The ppx pipeline uses OCR for markdown and while we can use PDFMiners LTChar for character-level bounding boxes that is a deterministic method that doesn't account for OCR noise. So to benchmark that against a pipeline that includes OCR noise we would not be isolating the correctness evaluation of the alignment.

Human bbox annotation requires a large coordinated effort. Previous work in this field can be seen with the DocLayNet dataset. They worked through 80,000 pages with 32 annotators in 3 months to draw boxes and picking labels. That work did not cover the granular complexity of token-level alignment between two sources.

Finally, if we were to evaluate correctness against Mistral or any other multi-modal OCR markdown we may be able to see how PaddleX stands against the industry standard, but that does not guarantee the correctness of the comparison itself.

## How can we frame it?

Reflecting back on our work reveals three meaningful contributions in a field that is yet to be explored:

1. A formal model for document-markdown alignment.
2. A composable pipeline.
3. The discovery of an inability to evaluate correctness

With the finding that token-level alignment ground truth does not exist, which is a necessary precondition for anyone who wants to benchmark this class of pipeline. We now have a structure of:

1. We build this alignment pipeline
2. We went to benchmark it and discovered the ground truth doesn't exist - here's precisely why.
3. We show that existing approaches all fail to resolve the problem for different reasons.
4. Failure mode analysis of the pipeline under those constraints.
5. We characterize what a proper ground truth dataset would require - which nobody has built.

## Failure mode analysis

Further research will be done to what failures we discovered. Such as `we identify N structural failure categories: multi-column layout, table regions, hyphenation across lines, OCR noise accumulation, figures, and supplementary regions`.

> - What breaks — the specific alignment behaviour that fails
> - Why it breaks — the structural cause (e.g. multi-column: PDFMiner reading order diverges from OCR reading order because OCR sees the page as an image)
> - How detectable it is — can you identify this failure programmatically without ground truth, or only by inspection?
>   That last point is actually interesting for the paper. Some failure modes are self-evident without ground truth — if the fuzzy match returns None, you know it failed. If the cursor has to fall back to position 0, that's a detectable signal of reading order divergence. Others (wrong span accepted within tolerance) are silent failures. The distinction between detectable and silent failures is a meaningful finding in itself, and maps directly to the open problems section — a proper benchmark would need to cover the silent failures specifically.
