## Designing a New Algorithm

I must at this point change my focus from the solution, such as presenting our findings, to the problem of the algorithm. The real problem we are facing is the _alignment_ problem between the Visual Tokens, and the markdown text.

To do this we must properly define the problem. Ken and I worked through a rigorous definition of the problem last week and made some considerable progress, but much is still left to be done. We ended up creating a graph representation of the document layout, finding an optimal tree structure such as a Steiner tree that encompasses the shortest path containing all nodes.

In the notes provided here we can see some of the concepts discussed. We broke the problem down into two sections:

1. Query Localization
2. Graph & Text Extraction

Where `Query Localization` focuses on finding the optimal location for a given string, and text extraction focuses on the alignment between Visual Tokens and markdown text.

## G and Text Extraction

If we are given a region of text we can define its graph in a way such that the root node `R` has v1, v2, ..., vn visual tokens, and each of vn has vn1, vn2, ..., vni sub regions. The tree then goes down recursively.

To find the optimal region of markdown for a given visual token we can do a two-step process:

1. Prune the set M of markdown regions with a sliding window to produce m candidates.
2. Iterate over m candidates with DP to find the optimal match.

We use pruning (sliding window) to find an independant argmax. For the windows we define s,e `subset` B, F. That is, the start and end of a window are within the beginning and finish of a region. This is a _locally optimized_ solution. Meaning that for the region R1 we find the top candidate(s), but that does not apply globally for all Ri.

To find a global opimum we need DP. Adding the pruning technique (previously missing in the LCS version of the algorithm) also ensures the following:

1. **Non-overlap**: two different `𝑣𝑖` pick the same markdown region
2. **Monotonic order**: `𝑣𝑖` appears above `𝑣𝑖+1` but its chosen span is after the span for `𝑣𝑖+1`.
3. **Coverage / gaps**: the “best” for some `𝑣𝑖` might jump far away because a common phrase repeats, leaving holes
4. **Merge/split reality**: sometimes one visual block corresponds to multiple md spans (or vice versa). Independent best-picks can’t coordinate that.

We can summarize as follows:

- **Sliding window**: “Where might this match?” (local evidence)
- **DP**: “Which set of matches is jointly optimal and consistent?” (global solution)
